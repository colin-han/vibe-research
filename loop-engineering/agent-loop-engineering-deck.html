PART · 01

定义与分类

先理清 vocabulary：什么是 agent loop，它和 workflow 有什么本质区别。
01 · 范式转移

为什么大家都在谈 loop

  • ▸从「一次性问答」到「循环执行」 — 2025 年起，社区共识从调 prompt 转向让模型在循环中自主工作。
  • ▸Agent = LLM 在循环中基于环境反馈使用工具 — 每轮：观察 → 思考 → 调用工具 → 拿结果 → 继续。
  • ▸Loop 让模型自主分解任务、迭代修正 — 不再需要人工把每一步写死。
  • ▸但 loop ≠ 银弹 — 自主性带来更高 token 成本与 compounding error 风险，必须工程化管控。
01 · Anthropic 分类法

Workflow 还是 Agent？先选对心智模型

WORKFLOW

预定义代码路径

LLM 和工具由你写的代码编排，流程路径可预测。适合任务清晰、可拆步骤的场景。

AGENT

LLM 自主循环

模型在循环中自主决定下一步、选哪个工具、何时停止。适合开放、不可预测的任务。

💡 二者统称 agentic systems。关键判断：任务复杂度 / 可预测性 够不够高，值得为它引入 LLM 的自主性。大多数应用只需 workflow，不必上 agent。
来源 · anthropic.com/research/building-effective-agents
01 · 可组合积木

五种 可组合 workflow 模式

PATTERN 1

Prompt Chaining

串行流水线，前一步输出喂给下一步，中间可加门控。

PATTERN 2

Routing

先分类输入，再分发到专门处理分支。

PATTERN 3

Parallelization

sectioning 并行拆分 / voting 投票聚合多个结果。

PATTERN 4

Orchestrator-Workers

lead agent 动态编排并行 subagent（见第 2 章）。

PATTERN 5

Evaluator-Optimizer

生成 → 评价 → 改进的显式 self-refine 循环。

原则

简单 > 框架

最成功的实现都不是用复杂框架，而是简单可组合模式搭出来的。

PART · 02

多 agent 与自我改进

多 agent 真的更强吗？loop 自我修正有效的硬条件是什么？
02 · orchestrator-worker 实证

多 agent：更强，但贵得多

PERFORMANCE
+90.2%

Opus 4 作 lead + Sonnet 4 subagent 的多 agent 系统，在 Anthropic 内部研究 eval 上比单 agent Opus 4 高 90.2%。

COST
~15× tokens

agent 约用 chat 的 4× token，多 agent 约用 15×。只有任务价值足够高、能覆盖性能溢价时才经济可行。

注：90.2% 为厂商在未公开内部 eval 上的相对提升（非绝对分）；token 成本为该系统实测、非普适定律。
来源 · anthropic.com/engineering/built-multi-agent-research-system
02 · 架构

多 agent 循环编排长什么样

lead agent 拆解任务 → subagent 并行调用工具 → 汇总。context 在各 subagent 间隔离，避免互相污染。
02 · 自我改进的硬界限

纯内在自纠无效——必须外部反馈

LLM 生成了一个回答，想再「自我修正」一轮
▼
仅凭内在能力修正（intrinsic self-correction）？
无任何外部信号
✗ 无效甚至更差
推理任务准确率不升反降
GSM8K / CommonsenseQA / HotpotQA 上，模型会把对的改成错的（Huang et al., ICLR 2024）。
✓ 需外部反馈
→ evaluator-optimizer pattern
verifier / PRM / 工具执行反馈 / RL 训练。o1、R1 等「有效自纠」全靠外部信号。
来源 · arxiv.org/abs/2310.01798 · anthropic.com/research/building-effective-agents
02 · test-time compute

「多想几步」不是越多越好

  • ▸不同方法的效果随 prompt 难度而变 — PRM 搜索 vs 自适应采样，谁更好取决于题有多难。
  • ▸compute-optimal 策略：按难度自适应分配算力 — 比 best-of-N 高 4×+ 效率（Snell et al., ICLR 2025）。
  • ▸中等难度题上，可胜过 14× 参数量大模型 — FLOPs 对等评测下，小模型靠 test-time compute 翻盘。
💡 loop 内的「多采样 / 多想几步」应按难度动态调节，而非无脑拉满——这就是 compute-optimal。

评测驱动开发闭环

没有 eval 的 loop 是黑盒。把 offline / online 评测串成迭代反馈环。
03 · LangSmith 闭环

offline ↔ online：迭代反馈环

部署到生产环境

💡 线上问题沉淀为线下测试用例，线下验证修复，线上再确认——eval 是闭环，不是一次性验收。
来源 · docs.smith.langchain.com/evaluation
03 · agent 评测对象

要评 trajectory，不是只看最终输出

  • ▸工具选择 — 选对了工具吗？
  • ▸参数格式 — 参数填对了吗？
  • ▸执行路径 — 走了几步？绕路了吗？
关键洞察

相同输出 ≠ 相同 agent

两个 agent 可能产出完全相同的最终答案，但一个读了 3 个文件、一个读了 30 个——成本、延迟、失败模式天差地别。只评输出会漏掉这些。

来源 · docs.smith.langchain.com/evaluation

PART · 04

落地与约束

从理论到工程：context engineering、防止失控、以及现实能力边界。
04 · 范式再迁移

Context Engineering 取代 prompt engineering

  • ▸定义 — 构建动态系统，以正确格式提供正确的信息、指令与工具。
  • ▸Anthropic 官方工程博客确认 — 从「找对措辞」到「配置正确 context」的范式转移。
  • ▸Karpathy 背书 — 公开站台「context engineering over prompt engineering」。
  • ▸Harrison Chase（LangChain） — prompt engineering 是 context engineering 的子集。
💡 大多数 agent 失败不再是模型失败，而是 context 失败。管理好 context window、隔离 sub-agent context、压缩与裁剪——这是 AI 工程师现在最重要的技能。
04 · 防止失控

把 loop 关进笼子

  • ▸显式终止条件（max iterations） — 不只靠「任务完成」判定，必须有迭代上限保持可控。
  • ▸Sandbox 环境大量测试 — agent 的自主性意味着更高成本，潜在 compounding error。
  • ▸Guardrails 兜底 — 限制工具权限、输出校验、成本 / 延迟告警。
  • ▸记住权衡 — 自主性 ↑ = 成本 ↑ + 误差累积风险 ↑。不是越自主越好。
来源 · anthropic.com/research/building-effective-agents
04 · 现实能力边界

METR：最强 agent 也只能可靠做几分钟的事

截至 2025-03，最强 frontier agent（Claude 3.7 Sonnet）虽能完成个别需要专家数小时的任务，但只能可靠完成几分钟级别的任务，无法稳健自动化哪怕是低技能的计算机工作（如远程行政助理），无法独立承接完整项目。
  • ▸第三方独立评测 — METR 非模型厂商，无美化动机，是 agent 能力的现实制衡。
  • ▸caveat — 这是 2025-03 baseline；2026 最新模型（Claude 4.x 等）的可靠时长上限大概率已上移，引用时务必标日期。
来源 · metr.org/blog/2025-03-19-measuring-ai-ability-to-complete-long-tasks

避坑：被否决的误区

05 · 五个 loop 神话

这些说法站不住脚

  • ✗intrinsic self-correction 能提升推理 — 实测在 GSM8K 等任务上不升反降，纯内在自纠不可靠。
  • ✗Self-Refine 单模型自反馈带来 ~20% 绝对提升 — 把 extrinsic 当 intrinsic 或夸大，未通过验证。
  • ✗多 agent 堆 token 就能变强（token 解释 80% 方差） — 该 claim 仅 1-2 被否决；性能不能只靠烧算力。
  • ✗agent 已能独立承接完整项目 — METR：目前只能可靠完成分钟级任务。
  • ✗test-time compute 通用万能 — 仅在中等难度区间有效，硬题/简单题收益有限。
💡 本次调研 25 条 claim 中 8 条被否决。引用任何「惊人数字」前，先确认它经过了独立验证——厂商内部数字尤其要打折。
